Relatório Técnico — Agente LLM com Tool de Previsão do Tempo

Candidato: Gabriel Willian  |  Data: Abril de 2026  |  Repositório: github.com/GabArtista/teste-climatempo

1. Solução

O agente utiliza uma arquitetura híbrida: a decisão de acionar a tool de clima é resolvida deterministicamente em 4 estágios (Stages 1–4). O LLM (qwen2.5:1.5b via Ollama) é invocado em dois contextos distintos: como classificador binário de fallback no Stage 5 (~5% dos casos) e, após a intenção ser confirmada, para formatar a resposta final com os dados reais retornados pela API.

Pipeline de 5 Estágios

Usuário → POST /api/v1/agent/chat
              │
              ▼
    [Stage 1] Keywords fortes → weather ──────────────────────────────┐
              │ (nenhuma match)                                          │
              ▼                                                          │
    [Stage 2] Frases fixas → weather ─────────────────────────────────┤
              │ (nenhuma match)                                          │
              ▼                                                          │
    [Stage 3] Exclusão pessoal → not_weather ──────────────────────────┤
              │ (nenhum match)                                           │
              ▼                                                          │
    [Stage 4] Score multi-sinal                                         │
              ├── score ≥ 0.55 → weather ────────────────────────────┤
              ├── score ≤ 0.20 → not_weather ──────────────────────────┤
              └── 0.20 < score < 0.55                                   │
                    │                                                    │
                    ▼                                                    │
    [Stage 5] LLM binário → weather / not_weather ─────────────────────┤
                                                                         │
                    ┌────────────────────────────────────────────────────┘
                    ▼  (intent = weather)
         _extract_city() ── Não encontrou ──→ Pedir cidade ao usuário
              │ Capital encontrada
              ▼
    WeatherService → GET api.open-meteo.com/v1/forecast
              │
              ▼
    LLM formata resposta com dados reais → Usuário

Stage 1 captura keywords explícitas (clima, previsão, temperatura, precipitação, forecast, weather). Stage 2 captura frases fixas como "como está o tempo", "vai chover", "vai fazer frio". Stage 3 exclui comentários pessoais que contêm palavras de clima mas não são queries ("tô com frio aqui", "ar-condicionado", "ela estava quente"). Stage 4 aplica scoring multi-sinal com thresholds (≥0.55 → weather; ≤0.20 → not_weather). Stage 5 usa o LLM como desempate nos casos intermediários.

3. Function Calling — Implementação

A WEATHER_TOOL segue o formato OpenAI. Schema compacto:

{"type": "function", "function": {"name": "get_weather_forecast", "parameters": {"type": "object", "properties": {"city": {"type": "string", "description": "Nome da capital brasileira"}, "forecast_days": {"type": "integer", "minimum": 1, "maximum": 7, "default": 3}}, "required": ["city"]}}}

Fluxo de execução: _classify_intent() determina se o prompt é weather. Se sim, _extract_city() localiza a capital por n-grams + regex contra capitals.json. O WeatherService chama GET api.open-meteo.com/v1/forecast e retorna os dados. O LLM recebe os dados reais como resultado de tool call e gera a resposta formatada ao usuário.

Variáveis coletadas: temperature_2m_max (°C máxima diária), temperature_2m_min (°C mínima diária), precipitation_sum (mm de chuva acumulada) — conforme especificado no desafio.

O único FP do modelo isolado foi o prompt "Qual é a previsão do tempo?": intenção correta, mas sem cidade especificada. No sistema completo, _extract_city() retorna None e o agente solicita a cidade ao usuário — o FP não se materializa como erro, resultando em Precision = 1.0 no sistema híbrido.

5. Qualidade de Dados

Anomalia detectada: capitals.json continha "Campo Grande - Rio Grande do Norte" (incorreto — a capital do RN é Natal). O sistema corrige automaticamente no startup via _validate_integrity(), substituindo pela entrada "Natal - Rio Grande do Norte". As anomalias detectadas ficam disponíveis em GET /api/v1/weather/data-quality.
